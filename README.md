# vpn-infra — Двухуровневая VPN-инфраструктура для обхода РКН/ТСПУ

Самохостинговое решение для стабильного доступа к заблокированным ресурсам. Трафик к заблокированным сайтам идёт через VPS за рубежом, незаблокированный трафик — напрямую. Управление через Telegram-бота.

---

## Ключевые возможности

- **Адаптивный failover**: 4 стека защиты, автоматическое переключение при блокировке
- **Split tunneling**: только заблокированный трафик идёт через VPS, остальное — напрямую
- **Без ручной настройки**: установка одним скриптом за 25–45 минут, 57 шагов
- **Telegram-бот**: управление инфраструктурой, добавление клиентов, алерты, мониторинг
- **Несколько клиентов**: поддержка AmneziaWG (обфускация) и WireGuard
- **Автообновление баз**: ежедневное обновление списков заблокированных адресов РКН в 03:00
- **Мониторинг**: Prometheus + Grafana на VPS, алерты в Telegram
- **Kill switch**: при падении туннеля трафик к заблокированным сайтам блокируется, не утекает через незащищённый канал
- **Идемпотентность**: скрипт установки можно запустить повторно, он продолжит с места остановки

```
  ▼
Домашний сервер — Ubuntu 24.04

  │    amneziawg  wg0  10.177.1.0/24
  │    wireguard  wg1  10.177.3.0/24
  │    nftables   kill switch + fwmark routing
  │    dnsmasq    split DNS + nftset blocked_dynamic
  │    watchdog   FastAPI :8080, управление стеками
  │    autossh-tier2  10.177.2.0/30 (SSH tun туннель к VPS)
  │
  │  Docker (172.20.0.0/24):
  │    telegram-bot      управление ботом
  │    xray-client-xhttp  SOCKS5 :1081 (experimental reality-xhttp)
  │    xray-client-vision SOCKS5 :1084 (stable reality-vision)
  │    cloudflared       CDN стек
  │    socket-proxy      доступ к Docker API
  │    node-exporter     метрики для Prometheus
  │
  │  Заблокированный трафик (nftables fwmark routing):
  │    blocked_static + blocked_dynamic → fwmark 0x1 → table marked → tun → VPS
  │    остальное → table vpn → eth0 (прямой интернет)
  │
  ├──[стек 1: CDN]            VLESS+WebSocket → Cloudflare Worker → VPS:8080
  ├──[стек 2: reality-vision] VLESS+REALITY+Vision, SNI www.microsoft.com → VPS:443
  ├──[стек 3: reality-xhttp]  VLESS+REALITY+XHTTP, SNI cdn.jsdelivr.net    → VPS:2083
  └──[стек 4: hysteria2]      QUIC+Salamander UDP 443
  ▼
VPS — Ubuntu 22.04/24.04 (за рубежом)
  │  3x-ui (панель + CDN inbound :8080)
  │  xray-reality-vision :443, xray-reality-xhttp :2083
  │  nginx (mTLS :8443), cloudflared
  │  prometheus, grafana, alertmanager, node-exporter
  │  hysteria2 (standalone)
  ▼
Интернет (заблокированные ресурсы)
```

## Как работает обход DPI/ТСПУ

Система использует **два независимых уровня защиты**, которые работают совместно.

### Уровень 1 — Split tunneling: какой трафик попадает на домашний сервер

Клиентское устройство (телефон, ноутбук) настраивается с `AllowedIPs` — списком подсетей, трафик к которым идёт через VPN-туннель (AmneziaWG/WireGuard), а не напрямую. Список содержит **≤ 500 агрегированных CIDR-записей** — только заблокированные адреса, не весь интернет. Незаблокированный трафик (Сбербанк, Госуслуги, отечественные сервисы) никогда не покидает вашу сеть.

### Уровень 2 — Маршрутизация на домашнем сервере: что слать на VPS

Трафик от клиентов, попавший на домашний сервер, делится на два потока с помощью `nftables`:

```
Трафик к адресу из blocked_static или blocked_dynamic
  → fwmark 0x1 → ip rule → table "marked" → tun-интерфейс → VPS → интернет

Трафик к незаблокированному адресу
  → table "vpn" → eth0 → прямой выход в интернет
```

`blocked_static` — набор правил nftables, загружается из баз при старте и обновляется ежедневно в 03:00.

`blocked_dynamic` — заполняется на лету: когда клиент запрашивает DNS-имя заблокированного домена, `dnsmasq` автоматически добавляет полученный IP в этот set (timeout 24h). Это покрывает сайты с часто меняющимися IP.

**Kill switch**: если тун-интерфейс к VPS упал — трафик к заблокированным адресам блокируется nftables (DROP), не утекает в открытый интернет.

### Обход глубокой инспекции пакетов (DPI/ТСПУ)

Система использует два принципиально разных подхода — они работают параллельно и дополняют друг друга.

#### Подход A — Туннель через VPS (4 стека)

Трафик между домашним сервером и VPS маскируется:

| Метод | Как обходит DPI |
|---|---|
| **AmneziaWG** (клиент → сервер) | Обфускация заголовков WireGuard: случайный junk в начале пакета — ТСПУ не распознаёт как WireGuard |
| **VLESS+REALITY+Vision** | Штатный TCP fallback на 443/tcp. Более предсказуемый и зрелый стек для автоматического failover |
| **VLESS+REALITY+XHTTP** | Экспериментальный резервный стек для ручных тестов. Оставлен в системе, но не участвует в автоматическом standby |
| **Cloudflare CDN** | Трафик через глобальную CDN Cloudflare — блокировка невозможна без отключения всего Cloudflare в России |
| **Hysteria2 + Salamander** | QUIC с обфускацией: выглядит как случайный UDP-поток |

Этот подход надёжен даже против IP-блокировок (IP скрыт за VPS), но добавляет задержку и зависит от VPS.

#### Подход B — Zapret: прямой обход DPI без VPS

Параллельно с туннельными стеками работает **zapret** (Linux-аналог GoodbyeDPI) — он перехватывает TCP-пакеты на уровне ядра и обманывает DPI, не отправляя трафик через VPS.

**Как это работает:**

```
Клиент (wg0/wg1) → домашний сервер
  ↓ nftables FORWARD chain
  ↓ nfqueue (очередь 200)
  ↓ nfqws — применяет DPI-bypass технику
  ↓ eth0 → напрямую в интернет (без VPS!)
```

DPI видит поток пакетов, по которому невозможно определить SNI (имя сайта):
- **fakedsplit**: ClientHello TLS разрезается посередине домена (`disc|ord.com`), перед ним отправляется фейковый пакет с истёкшим TTL. DPI получает фейк и обрезанный ClientHello — SNI нечитаем. Реальный сервер фейк не получает (TTL истекает раньше), соединение устанавливается.
- **fake+autottl**: фейковый ClientHello с TTL, рассчитанным автоматически так, чтобы он доходил только до DPI, но не до сервера.
- **split2, multisplit**: множественная фрагментация TLS handshake.

**Работает против**: SNI-based DPI (discord, Steam, YouTube), пассивного ТСПУ.
**Не работает против**: IP-блокировок (tg, Meta) — для них нужен туннель.

**Адаптивный подбор параметров (Thompson Sampling):**

Разные ISP по-разному реализуют DPI — один пресет не работает везде. Zapret автоматически подбирает лучшую технику для вашего провайдера:

1. При первом запуске — `quick probe`: тестирует топ-3 пресета + 1 случайный
2. Ночью (полный probe) — тестирует все **17 пресетов** последовательно
3. Каждый тест: реальное TCP+TLS подключение к `discord.com`, `store.steampowered.com`, `www.youtube.com` через nfqueue (SO_MARK + OUTPUT chain)
4. Результаты обновляют **Thompson Sampling** (Beta-distribution multi-armed bandit) — система учится, какой пресет работает на вашем ISP
5. При 3 подряд неудачах — экстренный re-probe

Текущий победитель на проверенном ISP: `fakedsplit+midsld+autottl+badsum` (пресет C01).

## Базы заблокированных адресов

Система использует несколько источников, которые объединяются и ежедневно обновляются:

| Источник | Что содержит |
|---|---|
| [antifilter.download](https://antifilter.download) | Реестр РКН: IP и подсети заблокированных ресурсов |
| [community.antifilter.download](https://community.antifilter.download) | Расширенный список от сообщества |
| [iplist.opencck.org](https://iplist.opencck.org) | Альтернативная база заблокированных IP |
| [zapret-info/z-i](https://github.com/zapret-info/z-i) | Официальный реестр РКН (домены + IP) |
| Статические AS-подсети | Google, Meta, Cloudflare, Akamai — агрегированные блоки |
| `manual-vpn.txt` | Ваши собственные адреса (добавляются через `/vpn add домен`) |
| `manual-direct.txt` | Адреса, которые всегда идут напрямую (`/direct add домен`) |

**Логика обновления (cron 03:00)**:
1. Загрузить каждый источник в отдельный кэш-файл
2. При недоступности источника — использовать предыдущую версию кэша (алерт если кэш старше 3 дней)
3. Объединить все источники, удалить дубликаты, агрегировать CIDR (до ≤ 500 записей для AllowedIPs)
4. Атомарно применить к nftables: `nft -f` одним файлом (без окна утечки)
5. Если новый список отличается от текущего — разослать обновлённые конфиги всем клиентам

**Валидация**: список принимается только если содержит > 100 записей и отличается от предыдущего не более чем на 50% (защита от повреждённого источника).

**dnsmasq + nftset**: помимо статических IP, система перехватывает DNS-запросы. Домены из `manual-vpn.txt` резолвятся через VPS DNS, полученные IP автоматически попадают в `blocked_dynamic` — даже если конкретный IP ещё не был в базе РКН.

## Требования

### Оборудование

| Компонент | Минимум | Рекомендуется |
|---|---|---|
| **Домашний сервер** | 4 ГБ RAM, 64 ГБ SSD, Ethernet, Ubuntu 24.04 | 8 ГБ RAM, 128 ГБ SSD |
| **Роутер** | Реальный (белый) IP, поддержка port forwarding | — |
| **VPS** | KVM, 1 vCPU, 1 ГБ RAM, Ubuntu 22.04/24.04 | 2 vCPU, 2 ГБ RAM |
| **Интернет дома** | Upload ≥ 5 Мбит/с | Upload ≥ 20 Мбит/с |

> **Важно**: роутер должен иметь реальный (белый) IP, не CGNAT. Домашний сервер должен быть подключён к роутеру по Ethernet.
> Формула максимального числа клиентов: `upload_Мбит ÷ 5`.

### Аккаунты и доступы

| Что | Зачем | Обязательно |
|---|---|---|
| Telegram-аккаунт | Управление ботом, алерты | Да |
| VPS с SSH-доступом (root) | Второй уровень туннеля | Да |
| Cloudflare аккаунт (бесплатный) | CDN-стек (наиболее устойчивый к блокировкам) | Нет |

## Быстрый старт

```bash
# На домашнем сервере (Ubuntu 24.04, под root или через sudo)
curl -fsSL https://raw.githubusercontent.com/Cyrillicspb/vpn-infra/master/install.sh | sudo bash
```

Скрипт запросит данные (VPS IP, Telegram токен и т.д.) на фазе 0, затем выполнит установку автоматически.
Поддерживаемый путь установки один: запуск непосредственно на домашнем сервере. Клиентские SSH-launcher скрипты удалены из репозитория.

## Ручные шаги после установки

После завершения скрипта потребуется сделать два ручных шага:

### 1. Проброс портов на роутере

Настройте port forwarding на IP домашнего сервера:

| Протокол | Внешний порт | Внутренний порт |
|---|---|---|
| UDP | 51820 | 51820 |
| UDP | 51821 | 51821 |

Инструкции для конкретных роутеров: [docs/INSTALL.md → Port Forwarding](docs/INSTALL.md#обязательный-ручной-шаг-port-forwarding)

### 2. Первый запуск Telegram-бота

Напишите боту `/start` — он автоматически зарегистрирует вас как администратора и покажет меню управления.

## Протоколы

### Туннельные стеки через VPS (адаптивный failover)

Перечислены от наиболее к наименее устойчивому. Watchdog переключает автоматически при деградации.

| # | Стек | Протокол | Порт VPS | Устойчивость | Скорость |
|---|---|---|---|---|---|
| 1 | **cdn** | VLESS+WebSocket через Cloudflare Worker | 8080 TCP | Максимальная | Умеренная |
| 2 | **reality-vision** | VLESS+REALITY+Vision, SNI www.microsoft.com | 443 TCP | Очень высокая | Высокая |
| 3 | **reality-xhttp** | VLESS+REALITY+XHTTP, SNI cdn.jsdelivr.net | 2083 TCP | Экспериментально | Нестабильно |
| 4 | **hysteria2** | QUIC+Salamander obfuscation | 443 UDP | Средняя | Максимальная |

> CDN-стек требует предварительной настройки Cloudflare Worker (опционально, но рекомендуется для максимальной устойчивости).

### Стек zapret — прямой DPI bypass (без VPS)

Работает **параллельно** с туннельными стеками. Не создаёт туннель — перехватывает пакеты через nfqueue и обманывает DPI на уровне ядра.

| Параметр | Значение |
|---|---|
| **Инструмент** | nfqws (zapret project) |
| **Техники** | fakedsplit, fake+autottl, split2, multisplit (17 пресетов) |
| **Выбор параметров** | Thompson Sampling (адаптивный, per-ISP) |
| **Тестовые хосты** | discord.com, store.steampowered.com, www.youtube.com |
| **Против чего работает** | SNI-based DPI, пассивный ТСПУ |
| **Против чего не работает** | IP-блокировки (для них — туннельные стеки) |
| **Задержка** | Минимальная (нет hop через VPS) |
